    Branching Strategy & Gitflow

    Walkthrough interactivo: implementar eliminar un ToDo, introducir un bug intencional y resolverlo con una rama bugfix/*.

    1. ¿Qué es Gitflow?

    Una estrategia de ramas con roles bien definidos para cada tipo de trabajo.

    Gitflow organiza el repositorio en torno a dos ramas de larga vida y varias ramas auxiliares de corta vida:

    RamaPropósitoOrigenMerge a
    main Código en producción. Cada commit es una versión liberada. — —
    develop Integración continua. Lo último estable que aún no se libera. main release/*
    feature/* Nueva funcionalidad (ej. eliminar un ToDo). develop develop
    bugfix/* Corrección de un bug detectado en develop antes de liberar. develop develop
    release/* Estabilización previa al lanzamiento. develop main + develop
    hotfix/* Parches urgentes en producción. main main + develop
    Caso de hoy: Vamos a agregar el endpoint DELETE /todos/{id} al proyecto Quarkus. Durante el desarrollo introduciremos un bug a propósito (eliminar siempre el ID 1) para demostrar cuándo y cómo se crea una rama bugfix/*.

    2. Estado inicial del repo

    Partimos de develop sincronizada con main.

    Antes de empezar cualquier trabajo, sincronizamos las ramas locales:

    # Asegurarnos de tener lo último
git checkout develop
git pull origin develop
    Regla de oro: nunca trabajes directamente en main ni en develop. Toda nueva funcionalidad o corrección vive en una rama auxiliar.

    3. Crear la rama feature/delete-todo

    Una rama por funcionalidad. Nombre descriptivo y en kebab-case.

    git checkout develop
git checkout -b feature/delete-todo

    Convenciones de nombrado típicas:

    • feature/<ticket>-<descripcion-corta> — ej. feature/TODO-42-delete-todo
    • Solo minúsculas, separadas por guiones.
    • Prefijo coherente con el tipo de trabajo (feature, bugfix, hotfix).

    4. Implementar el endpoint DELETE

    Agregamos el método al recurso REST… con un bug a propósito.

    Editamos TodoResource.java:

    src/main/java/.../TodoResource.java
    @Path("/todos")
public class TodoResource {

    @Inject TodoRepository repo;

    // ... GET, POST existentes ...

+    @DELETE
+    @Path("/{id}")
+    @Transactional
+    public Response delete(@PathParam("id") Long id) {
+        // 🐛 BUG INTENCIONAL: ignoramos el id recibido
+        //    y siempre borramos el ToDo con id = 1
+        boolean deleted = repo.deleteById(1L);
+        return deleted
+            ? Response.noContent().build()
+            : Response.status(404).build();
+    }
}
    🐛 Bug introducido a propósito: el método recibe id pero llama a repo.deleteById(1L). Cualquier petición DELETE /todos/{id} borra siempre el ToDo con ID 1, sin importar el valor del path.

    Hacemos commits pequeños y descriptivos:

    git add src/main/java/.../TodoResource.java
git commit -m "feat(todo): add DELETE /todos/{id} endpoint"

    5. Pull Request a develop

    Subimos la rama y abrimos un PR para revisión.

    git push -u origin feature/delete-todo

    En GitHub se abre el Pull Request con destino develop:

    Title:  feat(todo): add DELETE /todos/{id} endpoint
Base:   develop
Compare: feature/delete-todo

Description:
- Adds DELETE endpoint to TodoResource
- Returns 204 on success, 404 if not found
- Closes #42
    Code review: el revisor aprueba el PR porque la lógica "se ve bien". Los tests existentes pasan porque no había test específico que verificara que se elimina el ID correcto. (Foreshadowing…)

    6. QA detecta el bug 🐛

    En la rama develop, durante pruebas exploratorias, alguien nota algo raro.

    Reporte de QA:

    ### Steps to reproduce
1. POST /todos {"title":"comprar pan"}   → id=1
2. POST /todos {"title":"sacar basura"}  → id=2
3. POST /todos {"title":"estudiar"}      → id=3
4. DELETE /todos/3                       → 204 ✅
5. GET /todos                            → [{id:2,...}, {id:3,...}]

### Expected
El ToDo con id=3 debe haberse borrado.

### Actual
Se borró el ToDo con id=1. 😱
    Decisión: el bug está en develop, todavía no llegó a main. Por lo tanto se corrige con una rama bugfix/* (no hotfix/*, que sería para producción).

    7. Crear bugfix/delete-todo-wrong-id

    Rama corta, enfocada en una sola corrección.

    git checkout develop
git pull origin develop
git checkout -b bugfix/delete-todo-wrong-id

    Antes de tocar el código de producción, escribimos un test que falle y exponga el bug. Esto nos garantiza que la corrección efectivamente arregla el problema y evita regresiones futuras.

    src/test/java/.../TodoResourceTest.java
    +    @Test
+    void deleteShouldRemoveTheRequestedId() {
+        Long id1 = createTodo("a");
+        Long id2 = createTodo("b");
+
+        given().when().delete("/todos/" + id2)
+              .then().statusCode(204);
+
+        given().when().get("/todos")
+              .then().body("id", hasItem(id1.intValue()))
+                    .body("id", not(hasItem(id2.intValue())));
+    }
    ./mvnw test
# → FAILED: deleteShouldRemoveTheRequestedId  ❌
#   Expected id=2 to be removed, but id=1 was deleted instead.
    El test rojo confirma que reproducimos el bug. Ahora podemos arreglarlo con confianza.

    8. Aplicar el fix

    Cambio mínimo: usar el id recibido en lugar de la constante.

    src/main/java/.../TodoResource.java
        @DELETE
    @Path("/{id}")
    @Transactional
    public Response delete(@PathParam("id") Long id) {
-        // 🐛 BUG INTENCIONAL: ignoramos el id recibido
-        //    y siempre borramos el ToDo con id = 1
-        boolean deleted = repo.deleteById(1L);
+        boolean deleted = repo.deleteById(id);
        return deleted
            ? Response.noContent().build()
            : Response.status(404).build();
    }
    ./mvnw test
# → Tests run: 12, Failures: 0  ✅

    Commits del bugfix:

    git add src/test/java/.../TodoResourceTest.java
git commit -m "test(todo): expose delete-by-id bug"

git add src/main/java/.../TodoResource.java
git commit -m "fix(todo): use path id instead of hardcoded 1"

    9. Merge del bugfix a develop

    PR pequeño, enfocado, fácil de revisar.

    git push -u origin bugfix/delete-todo-wrong-id

    El PR del bugfix incluye:

    • Un test que reproduce el bug (rojo antes del fix).
    • El cambio mínimo que arregla el comportamiento.
    • Referencia al issue/reporte de QA.

    Tras el merge, el grafo del repo queda así:

    Borramos la rama remota y local (higiene):

    git checkout develop
git pull origin develop
git branch -d bugfix/delete-todo-wrong-id
git push origin --delete bugfix/delete-todo-wrong-id

    10. De develop a main

    Cuando develop está estable, se libera vía release/*.

    git checkout develop
git checkout -b release/1.1.0
# Bump versions, changelog, smoke tests…
git commit -am "chore(release): 1.1.0"

# Merge a main y tag
git checkout main
git merge --no-ff release/1.1.0
git tag -a v1.1.0 -m "Release 1.1.0"

# Merge de regreso a develop por si hubo fixes en release
git checkout develop
git merge --no-ff release/1.1.0
    Resultado: el feature delete a ToDo llegó a producción sin el bug, gracias a que la estrategia de ramas nos dio un espacio (develop + bugfix/*) para detectarlo y corregirlo antes del release.

    Lecciones clave

    • Ramas auxiliares cortas y enfocadas. Una rama, un propósito.
    • Test antes que fix. Un test rojo es la mejor prueba de que reproduces el bug.
    • bugfix/* vs hotfix/*. El destino del bug determina la rama, no la urgencia subjetiva.
    • Code review no sustituye a los tests automáticos: ambos atrapan cosas distintas.
    • develop es tu red de seguridad. Mientras el bug esté ahí y no en main, los usuarios siguen felices.
