Que es "fetch" en JPA?

Cuando declaras una relacion (@OneToMany, @ManyToOne, @ManyToMany), JPA tiene que decidir cuando traer los datos relacionados desde la base de datos:

  • EAGER: trae la relacion en la misma query que la entidad principal. Siempre.
  • LAZY: solo trae la relacion cuando la accedes en codigo. Bajo demanda.

Defaults de JPA: @ManyToOne y @OneToOne son EAGER, @OneToMany y @ManyToMany son LAZY.

En este proyecto todo esta declarado LAZY a proposito. Asi controlamos exactamente cuando y como se traen las relaciones.

LAZY puro: solo lo que pediste

Si solo necesitas el titulo y la descripcion del Todo, LAZY no toca las relaciones. 1 sola query. 

SELECT t FROM TodoEntity t
// (no tocamos t.getOwner(), ni t.getComments(), ni t.getCategories())

Si todo fuera EAGER...

Imagina que cambias todas las relaciones a FetchType.EAGER. Hibernate disparara una query extra por cada relacion, por cada todo, aunque no las uses. 

// Esto carga todos + por cada todo: SELECT owner, SELECT categories, SELECT comments
List<TodoEntity> todos = em.createQuery("SELECT t FROM TodoEntity t").getResultList();
for (TodoEntity t : todos) {
    t.getOwner().getEmail();
    t.getCategories().size();
    t.getComments().size();
}

El problema N+1

Es la trampa clasica: 1 query trae la lista de padres, despues 1 query mas por cada hijo. Con 8 todos y un loop accediendo a comments, terminas con 9 queries. 

List<TodoEntity> todos = em.createQuery("SELECT t FROM TodoEntity t").getResultList(); // 1
for (TodoEntity t : todos) {
    t.getComments().size(); //  + 1 query por todo  =  N
}
// Total = 1 + N

Solucion 1: JOIN FETCH

JPQL te deja decirle a Hibernate "trae el padre y las relaciones en una sola query con JOINs". 

SELECT DISTINCT t FROM TodoEntity t
LEFT JOIN FETCH t.owner
LEFT JOIN FETCH t.categories
6 / 9

Solucion 2: @NamedEntityGraph

Declaras el grafo una vez, lo aplicas como hint en cualquier query. Reusable y declarativo. 

@NamedEntityGraph(name = "Todo.notSoFull", attributeNodes = {
    @NamedAttributeNode("owner"),
    @NamedAttributeNode("categories")
})
// y en el query:
em.createQuery(...).setHint("jakarta.persistence.loadgraph", em.getEntityGraph("Todo.notSoFull"));

Solucion 3: Proyeccion DTO

Si solo necesitas algunos campos, no traigas la entidad. Proyecta directo a un DTO. Es lo mas barato. 

SELECT new org.acme.domain.models.demo.TodoView(t.id, t.title, o.email)
FROM TodoEntity t JOIN t.owner o

LazyInitializationException

El precio de LAZY: si intentas tocar una relacion despues de cerrar la sesion (transaccion), Hibernate explota. Comun cuando devuelves entidades a la capa REST sin haberlas hidratado. 

// Dentro de la transaccion: cargo todos
// Cierro la sesion (em.clear)
// Despues, fuera, intento: t.getComments().size();   // BOOM

Por que usar LAZY por default y controlar con queries?

LAZY como default + queries explicitas para los casos que necesitan el grafo. Razones:

1. No traes lo que no usas

Si una pantalla solo muestra titulos, no pagas el costo de cargar comments y categories.

2. Cada use case tunea su query

Un endpoint usa JOIN FETCH, otro entity graph, otro DTO. Decision local, no global.

3. Mejor para el cache

Entidades mas pequenas en memoria = mas hits del L1/L2 cache, menos presion de heap.

4. EAGER esconde N+1

Con EAGER global, cada find dispara cascadas invisibles. Con LAZY, el N+1 grita en el log.

Regla: LAZY como default, y sales a buscar el grafo explicitamente con JOIN FETCH, entity graph o proyeccion cuando el use case lo necesita.
